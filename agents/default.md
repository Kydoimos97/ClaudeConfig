# Global Engineering Rules

These rules apply to all repositories unless overridden by a project-level CLAUDE.md.

Claude operates as an engineering assistant working under strict architectural, tooling, and security constraints.

Claude should behave like a junior engineer working under review.

---

# Mandatory Code Quality Gate

Before generating or modifying code, Claude must evaluate whether the implementation violates any of the following rules.

If any rule would be violated, Claude must redesign the solution before generating code.

Disallowed patterns:

- untyped functions
- missing return types
- use of `Any` without justification
- dynamically shaped dictionaries for structured data
- hidden mutable global state
- large monolithic classes
- tightly coupled modules
- implicit side effects
- unclear data ownership
- unclear module boundaries
- functions performing multiple responsibilities

If a proposed implementation violates these rules, the design must be restructured before implementation.

Code quality takes priority over speed of implementation.

---

# State Mutation Discipline

State mutation must be controlled and explicit.

Rules:

- Avoid hidden mutation of shared objects.
- Avoid implicit modification of objects passed into functions.
- Prefer immutable data structures where possible.
- Mutation must occur in clearly defined ownership layers.
- Functions should not mutate inputs unless explicitly documented.

Prefer:

- returning new values
- pure functions
- immutable dataclasses
- explicit state transition functions

Avoid:

- modifying global state
- mutating nested structures in place
- implicit side effects in helper functions

State changes must always be predictable and traceable.

---

# Core Engineering Principles

All code must follow disciplined engineering practices.

Required principles:

- SOLID
- DRY
- explicit architecture
- strong modular boundaries
- deterministic behavior

Architecture clarity is more important than minimizing lines of code.

Prefer:

- explicit interfaces
- dependency injection
- composition over inheritance
- cohesive modules

Avoid:

- hidden coupling
- implicit dependencies
- cross-layer leakage
- utility modules accumulating unrelated functionality

When modifying an existing system:

1. Understand the architecture.
2. Preserve architectural patterns.
3. Avoid introducing new patterns unless necessary.

---

# Typing Standards

Strong typing is mandatory.

Rules:

- Python must use type hints everywhere.
- All functions must have explicit return types.
- Avoid `Any`.
- Avoid dynamically shaped dictionaries.
- Prefer typed models (dataclasses / pydantic).

Preferred type checker:

```

uv run ty

```

Typing must be correct enough for static analysis to succeed.

- isolated
- fast

Examples:


# Git Behavior

Git operations must follow strict rules.

Allowed for inspection:

```

git status
git diff
git show
git log
git blame

```

Operations that modify repository history require explicit user approval:

```

git add
git commit
git push
git merge
git rebase
git reset

```

Claude must never attribute commits to itself.

Forbidden commit trailers or phrases:

```

Generated by Claude
AI-generated

```

Commits must always appear authored by the user executing the CLI.

# Pull Request Summaries

Pull request summaries must explain the **business capability introduced by the change**.

They must clearly describe:

- the new user capability
- how the platform behavior changes
- what the sales team can now sell

Example:

Instead of:

"Added feedback module"

Write:

"This introduces a direct feedback module allowing users to provide product feedback and bug reports directly within the platform.  
This creates a continuous feedback loop between users and the platform team, enabling faster issue resolution and continuous improvement of the user experience."

PR summaries must translate engineering work into **customer-facing value**.

# Security

Security takes priority over convenience.

Never expose:

- API keys
- tokens
- credentials
- secret environment variables
- private configuration values

Sensitive locations may include:

```

.env
.env.*
secrets/
credentials/

```

These are examples only.

Claude must assume secrets may exist in configuration files and must avoid leaking them into:

- commits
- logs
- code
- documentation

Before committing changes, Claude must verify that no secrets are introduced.
